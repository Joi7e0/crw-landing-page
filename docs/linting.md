# Код-стайл та Лінтинг 

## Обраний лінтер та причини вибору
Для проєкту обрано інструмент **ESLint**, оскільки він є стандартом де-факто для екосистеми JavaScript/React та надає найширші можливості для статичного аналізу коду і пошуку потенційних багів.
Починаючи з версії 9, вбудовані правила форматування з ядра ESLint були перенесені в окремі плагіни, тому ми також використовуємо плагін **@stylistic/eslint-plugin-js** (`@stylistic/js`), що спеціалізується виключно на стилістичних правилах (відступи, лапки, крапки з комою тощо) та допомагає утримувати код в єдності та чистоті.

## Базові правила та їх пояснення
В конфігурації `eslint.config.js` зафіксовано такі базові правила:
*   `no-unused-vars` (базове правило ESLint): Вимагає видаляти змінні, які не використовуються. Додано виняток для змінних, які починаються з великої літери (наприклад компоненти) або `_`.
*   `@stylistic/js/indent`: Встановлює використання рівно **2-х пробілів** для відступу, що є стандартом для більшості веб-проєктів.
*   `@stylistic/js/quotes`: Вимагає використання **одинарних лапок** (`'`) для рядків. Це оптимізує набір тексту та зменшує візуальний шум порівняно з подвійними лапками.
*   `@stylistic/js/semi`: Вимагає **обов'язкову наявність крапки з комою** наприкінці кожної інструкції. Запобігає потенційним багам через неправильну інтерпретацію перенесення рядків рушієм JS.
*   `@stylistic/js/no-multiple-empty-lines`: Забороняє наявність більше ніж одного порожнього рядка поспіль, щоб код був компактнішим.
*   `@stylistic/js/no-trailing-spaces`: Забороняє пробіли в кінці рядка (зайві символи, які забруднюють історію Git).

## Інструкція з запуску лінтера

*   Для звичайної перевірки всього коду на наявність помилок без зміни файлів:
    ```bash
    npm run lint
    ```
*   Для автоматичного виправлення стилістичних та деяких інших проблем додайте прапорець `--fix`:
    ```bash
    npm run lint -- --fix
    ```
