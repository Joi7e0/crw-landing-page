1.	ВСТУП
Дана робота присвячена розробці графічного інтерфейсу користувача для налаштування динамічної маршрутизації на маршрутизаторах Cisco з використанням SSH-з'єднання. Метою проекту є створення крос-платформного, веб-орієнтованого графічного інтерфейсу, призначеного для того, щоб дозволити адміністраторам мереж зручно та безпечно налаштовувати динамічні протоколи маршрутизації, такі як OSPF, EIGRP або BGP, на обладнанні Cisco, мінімізувавши необхідність прямого використання командного рядка (CLI). Реалізація інтерфейсу передбачає використання сучасного стеку веб-технологій, зокрема Python з бібліотекою Eel а також JavaScript, HTML та CSS для створення інтуїтивної та функціональної користувацької частини.
Актуальність та необхідність даної розробки обумовлена низкою існуючих проблем у сфері адміністрування корпоративних мереж. Насамперед, це складність та схильність до помилок ручного налаштування через CLI, що вимагає від адміністратора, особливо початківця, глибокого знання синтаксису конкретних команд Cisco IOS. По-друге, на ринку відчувається брак доступних спеціалізованих інструментів: професійні рішення на кшталт Cisco DNA Center є дорогими та надлишково складними для невеликих та середніх мереж, тоді як існуючі безкоштовні аналоги часто мають обмежену функціональність або застарілий інтерфейс. Крім того, існує потреба в централізованому управлінні конфігурацією кількох пристроїв з одного вікна, що значно скорочує час на виконання рутинних операцій. Важливим аспектом також є забезпечення безпеки та стандартизації процесу налаштування. Використання SSH як транспорту гарантує захищене з'єднання, а графічний інтерфейс дозволяє впровадити валідацію вхідних даних та використання шаблонів, що запобігає появі людських помилок у конфігураціях.
Ключовими характеристиками розроблюваного програмного продукту є: інтуїтивний веб-інтерфейс з адаптивним дизайном, що включає форми, поля введення та візуалізацію статусу пристроїв; безпечне підключення до обладнання через вбудований SSH-клієнт, реалізований за допомогою бібліотек Python (Paramiko або Netmiko); механізм динамічної генерації конфігурацій, який автоматично перетворює задані користувачем параметри у відповідні команди Cisco IOS та відправляє їх на маршрутизатор. Завдяки архітектурі, заснованій на веб-технологіях, продукт є крос-платформним і працюватиме на будь-якій сучасній операційній системі (Windows, Linux, macOS) у звичайному браузері. Модульний підхід до розробки закладає основу для майбутнього розширення функціоналу, такого як додавання підтримки нових протоколів маршрутизації або обладнання інших виробників.
2.	ФУНКЦІОНАЛЬНІ ВИМОГИ ДО ПРОЄКТУ
2.1.	 Загальні вимоги
Застосунок є настільною програмою, що повинна коректно функціонувати на операційній системі Windows, Linxu, MacOs. Для забезпечення широкої доступності та зручності інтерфейс користувача має бути адаптивним та коректно відображатися на роздільних здатностях екрану, починаючи від 1366x768 пікселів.
Ключовою для роботи проекту є можливість створення та експорту текстових конфігураційних файлів для маршрутизаторів Cisco. Замовник приділяє особливу увагу коректності генерованих команд та локальній роботі програми. Ведення лог-файлу дій користувача  є бажаним функціоналом для налагодження та аналізу.
Продукт, що розробляється, є настільною програмою, тому вимоги до кількості одночасних користувачів, протоколу HTTPS та обробки веб-запитів RPM не застосовуються. Продуктивність визначається наступними локальними критеріями:
•	90% операцій генерації конфігурації мають виконуватися за 3000 мс або менше.
•	99% операцій відкриття/збереження проекту мають виконуватися за 5000 мс або менше.
•	Загальний час відгуку інтерфейсу на будь-яку дію користувача не повинен перевищувати 1000 мс.
Програма має бути оптимізована для роботи з обмеженнями системних ресурсів CPU, RAM типових настільних комп'ютерів.

2.	Архітектура системи 
2.1.	Загальна концепція архітектури
Проект реалізується за шаблоном Desktop Web Application, який поєднує  Python-бекенд із веб-інтерфейсом. Така архітектура розділяє систему на два шари:
	бекенд відповідає за всю бізнес-логіку, валідацію даних, генерацію команд Cisco IOS та управління конфігурацією; 
	фронтенд (HTML/CSS/JS) забезпечує інтуїтивний та адаптивний інтерфейс для користувача. Зв'язок між шарами забезпечує легка бібліотека Eel, яка створює міст для двосторонньої комунікації.

2.2.	Архітектурний стиль програми
Структура модулів реалізована за модульним принципом для чіткого розділення обов'язків та простоти розширення. На скриншоті (зображення 3.1) зображено структуру проєкту організація файлів відповідає наступній логіці:
 
Зоб 3.1 Структура модулів відповідно до реалізованої файлової 
Архітектура проєкту побудована за модульним принципом настільного веб-застосунку з локальним виконанням. Код поділений на три логічні частини: backend, web і tests. У каталозі backend зосереджена вся бізнес-логіка програми на Python: main.py є точкою входу та відповідає за запуск застосунку і зв’язок з інтерфейсом, generate.py формує конфігураційні команди Cisco IOS, validate.py виконує перевірку коректності введених користувачем даних, protocols.py містить реалізацію логіки протоколів маршрутизації, а каталог config використовується для зберігання налаштувань і допоміжних параметрів. Каталог web містить клієнтську частину застосунку, реалізовану на HTML, CSS і JavaScript, яка відповідає за відображення інтерфейсу, збір введених даних і ініціювання викликів функцій бекенду. Взаємодія між фронтендом і бекендом відбувається напряму без окремого серверного шару, що дозволяє працювати автономно. Каталог tests містить модульні тести для перевірки коректності роботи основної логіки. Така архітектура забезпечує чітке розділення відповідальностей, простоту підтримки та можливість розширення функціональності без зміни базової структури проєкту.
 
3.	СНОВНІ ФУНКЦІОНАЛЬНІ МОДУЛІ
3.1.	Базова IP-конфігурація 
На цьому етапі додаються основні мережеві налаштування маршрутизатора. Без призначених IP-адрес на інтерфейсах маршрутизатор не може комунікувати в мережі, тому даний крок є фундаментом для всіх подальших налаштувань.
3.2.	 Протокол маршрутизації 
Даний крок відповідає за налаштування динамічного протоколу маршрутизації, такого як OSPF, RIP, EIGRP, BGP або IS-IS. Його призначення полягає в автоматичному обміні інформацією про мережі між маршрутизаторами та формуванні таблиць маршрутизації. 
3.3.	IP Multicast-Routing
На цьому етапі вмикається підтримка групового (multicast) трафіку. Така функціональність необхідна для потокового відео, відеоконференцій та інших сервісів, де одна копія даних передається одразу багатьом одержувачам
3.4.	 Telephony Service
Цей крок призначений для налаштування телефонних пристроїв на маршрутизаторі з метою інтеграції VoIP-телефонії в мережеву інфраструктуру. 
3.5.	Конфігурація безпеки 
На цьому етапі виконується налаштування механізмів захисту доступу до маршрутизатора. Це включає встановлення паролів для різних рівнів доступу та активацію захищеного віддаленого підключення. 
 
3.6.	DHCP-сервер
Даний крок відповідає за налаштування DHCP-сервера, який автоматично розподіляє IP-адреси клієнтським пристроям у мережі. Це дозволяє значно спростити управління адресним простором та зменшити ймовірність помилок при ручному налаштуванні.
3.7.	NAT-конфігурація
На цьому етапі налаштовується трансляція мережевих адрес, що забезпечує доступ пристроїв із приватних мереж до зовнішніх ресурсів через публічну IP-адресу. 
3.8.	SNMP та моніторинг
Цей крок призначений для налаштування засобів моніторингу стану маршрутизатора. Він включає активацію протоколу SNMP та системного логування для віддаленого спостереження за роботою обладнання.
3.9.	Збереження або експорт конфігураці
Заключний етап відповідає за формування повної конфігурації маршрутизатора на основі всіх попередніх налаштувань та її збереження у файл. Це дозволяє експортувати готову конфігурацію для подальшого використання або завантаження на реальний мережевий пристрій. У користувацькому інтерфейсі відображається згенерований конфігураційний текст, а також доступні функції копіювання та збереження файлу.
